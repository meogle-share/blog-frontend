# 커밋 생성

아래 컨벤션에 따라 현재 변경사항을 커밋한다.

## 절차

1. `git status`와 `git diff`로 변경사항 확인
2. `git log --oneline -5`로 최근 커밋 스타일 참고
3. 아래 컨벤션에 따라 커밋 메시지 작성
4. 변경사항을 스테이징하고 커밋

## 커밋 단위
- 하나의 커밋 = 하나의 논리적 변경 (한 문장으로 설명 가능해야 함)
- 리팩터링과 기능 변경은 분리
- 변경사항이 여러 논리적 단위로 나뉘면 커밋을 분리한다

## 제목 형식

`type(scope): 설명`

- scope: 기능 영역 (`auth`, `blog`, `dashboard`, `ui` 등). 전체 범위면 생략
- 설명: "무엇을 했는가"를 한글로 간결하게
- **제목 전체 72자 이내** — `git log --oneline`에서 잘리지 않는 한계

### type 선택 기준

| type | 기준 | 예 |
|------|------|----|
| `feat` | 사용자에게 새 기능이나 동작 변경을 제공 | 로그인 페이지 추가, 게시글 목록 무한스크롤 |
| `fix` | 기존 기능의 버그 수정 | 다크모드 전환 시 깜빡임 수정 |
| `refactor` | 동작 변경 없이 코드 구조 개선·제거 | 미사용 컴포넌트 제거, 상태 관리 구조 정리 |
| `chore` | 빌드·설정·의존성 등 비즈니스 로직 외 변경 | 패키지 업데이트, CI 설정 |
| `docs` | 문서만 변경 | README 수정, 스타일 가이드 추가 |
| `style` | UI/스타일만 변경 (레이아웃, 색상, 간격 등) | 카드 간격 조정, 반응형 브레이크포인트 수정 |

> **판단 팁**: "이 커밋을 되돌리면 사용자가 눈치채는가?" → Yes면 `feat`/`fix`, No면 `refactor`/`chore`

## 본문
- 제목만으로 "왜"가 충분히 전달되면 생략
- 필요한 경우: 기존 동작이 바뀌는 이유, 선택하지 않은 대안, 여러 파일에 걸친 맥락
- "왜 이 변경이 필요한가"에 집중 — diff를 반복하지 않는다

## 예시

```
# 좋은 예
feat(auth): 로그인 페이지 추가
fix(blog): 게시글 상세 페이지에서 마크다운 렌더링 깨지는 오류 수정
style(ui): 카드 컴포넌트 반응형 레이아웃 적용
chore: ESLint 설정 업데이트
refactor: 미사용 컴포넌트 제거

# 본문이 필요한 경우
feat(auth): 토큰 갱신 방식 변경

기존에는 access token 만료 시 로그인 페이지로 리다이렉트했으나,
axios interceptor에서 refresh token으로 자동 갱신하도록 변경한다.
UX 개선과 보안 사이의 균형을 위해 refresh token 유효기간은
7일로 제한한다.

# 나쁜 예 → 개선
feat(auth): 로그인 UI 추가 및 인증 상태 관리 개선
→ feat(auth): 로그인 페이지 추가  (상태 관리 개선은 별도 커밋)

feat: 사용하지 않는 컴포넌트 제거
→ refactor: 미사용 컴포넌트 제거  (feat 아님, 기능 추가가 아니라 정리)

```
